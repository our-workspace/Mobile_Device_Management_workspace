# 📱 Mobile Device Management (MDM) System - دليل الإعداد والتشغيل

هذا الدليل أُعد خصيصًا لشرح متطلبات بناء، تشغيل، وربط مكونات مشروع نظام إدارة الأجهزة (MDM). يتكون المشروع من ثلاثة أجزاء رئيسية:
1. **الخادم (Backend)**: مبني بـ Node.js و Fastify و Prisma.
2. **لوحة التحكم (Dashboard)**: مبنية بـ React و Vite.
3. **تطبيق الأندرويد (Agent)**: مبني بـ Kotlin للعمل كعميل في خلفية الأجهزة.

---

## 🛠️ المتطلبات الأساسية (Requirements)

قبل البدء، يجب التأكد من تثبيت البرامج التالية على جهازك أو سيرفرك:
- **Node.js** (الإصدار 18 أو أحدث)
- **PostgreSQL**: قاعدة البيانات الأساسية للمشروع.
- **Redis**: لإدارة الجلسات السريعة وحالة الاتصال (WebSockets).
- **Android Studio** أو **JDK 17**: لبناء وتعديل تطبيق الأندرويد.
- مدير الحزم **npm** أو **yarn**.

---

## 🏗️ 1. إعداد وتشغيل الخادم (mdm-backend) وكيفية إنشاء قاعدة البيانات

هذا الجزء هو العقل المدبر للنظام الذي يربط بين الأجهزة ولوحة التحكم.

### خطوات الإعداد وإنشاء قاعدة البيانات:
1. افتح سطر الأوامر وانتقل إلى مجلد الخادم:
   ```bash
   cd mdm-backend
   ```
2. قم بتثبيت الحزم (Dependencies):
   ```bash
   npm install
   ```
3. إعداد متغيرات البيئة:
   قم بإنشاء ملف `.env` (أو عدّل الموجود) في مسار `mdm-backend` وضع فيه المتغيرات التالية (مع تعديل البيانات لتناسب إعداداتك):
   ```env
   # إعدادات قاعدة البيانات PostgreSQL (ضع اسم المستخدم، كلمة المرور واسم قاعدة البيانات)
   DATABASE_URL="postgresql://username:password@localhost:5432/mdmdb?schema=public"

   # إعدادات Redis
   REDIS_URL="redis://localhost:6379"

   # المنفذ ومفتاح التشفير الخاص بالتأمين (JWT)
   PORT=3000
   JWT_SECRET="YOUR_SUPER_SECRET_KEY"
   ```
4. **تأسيس وإنشاء قاعدة البيانات باستخدام Prisma:**
   من خلال سطر الأوامر، نقوم بدفع المخطط (Schema) المتواجد بالكود إلى قاعدة البيانات الفارغة لإنشاء جميع الجداول المطلوبة (مثل Device, Command, Heartbeat وغيرها):
   ```bash
   # تجهيز Prisma Client
   npx prisma generate
   
   # رفع وتطبيق المخطط على قاعدة البيانات لإنشاء الجداول
   npx prisma db push
   
   # ملاحظة: للاستخدام في بيئة التطوير يمكنك استخدام المايكريشن
   # npx prisma migrate dev --name init
   ```
5. تشغيل السيرفر السحابي:
   ```bash
   npm run dev
   ```
   *سيعمل السيرفر الآن على المنفذ `3000` (مثلًا: `http://localhost:3000`).*

## 🖥️ 2. إعداد وتشغيل واجهة المشرف (mdm-dashboard)

1. افتح نافذة أوامر جديدة (Terminal) وانتقل لمجلد الداشبورد:
   ```bash
   cd mdm-dashboard
   ```
2. تثبيت الحزم المطلوبة:
   ```bash
   npm install
   ```
3. إعداد الاتصال بالخادم:
   تأكد أن الداشبورد يتصل بـ `localhost:3000` (أو عنوان IP السيرفر الخاص بك). يمكنك تعديل ملف `.env` داخل مجلد الداشبورد بحيث يحتوي على مسارات الـ API والـ WebSockets الموجهة للخادم:
   ```env
   VITE_API_BASE_URL=http://localhost:3000/api/v1
   VITE_WS_BASE_URL=ws://localhost:3000/ws/dashboard
   ```
4. تشغيل للمعاينة المحلية:
   ```bash
   npm run dev
   ```
   *سيقوم Vite بتشغيل الداشبورد على منفذ محلي وعرض مساره في واجهة الأوامر (غالبًا `http://localhost:5173`).*

## 📱 3. تجهيز وبناء تطبيق الأندرويد (mdm-agent)

لكي يعمل التطبيق على الأجهزة ويرتبط بالسيرفر بشكل سليم، يجب توجيهه للـ IP الحقيقي الخاص بالسيرفر. (هام: لا تستخدم localhost أو 127.0.0.1 لأن النظام سيعتبر أن السيرفر موجود على نفس الهاتف، ولن يتصل بجهازك).

1. افتح مجلد `mdm-agent` باستخدام **Android Studio**.
2. **تجهيز الاتصال بالسيرفر:**
   قم بتغيير عنوان IP ليشير إلى سيرفر التشغيل، وذلك بتعديل روابط السيرفر داخل الكود (مثل `ApiClient.kt` و `WebSocketManager.kt`) أو داخل الخصائص البيئية. 
   مثال للإصلاح:
   ```kotlin
   val API_BASE_URL = "http://192.168.1.100:3000/api/v1"
   val WS_URL = "ws://192.168.1.100:3000/ws/agent"
   ```
3. قم بعمل مزامنة لبيئات Gradle بالضغط على (Sync Project with Gradle Files).
4. بناء التطبيق وإنشاء ملف الـ APK:
   * **طريقة 1:** من القائمة العلوية من Android Studio: `Build > Build Bundle(s) / APK(s) > Build APK(s)`
   * **طريقة 2:** باستخدام سطر الأوامر وأنت داخل مجلد مسار mdm-agent:
     ```bash
     .\gradlew assembleDebug
     ```
     *(لبيئات اللينكس/الماك استخدم `./gradlew assembleDebug`)*
5. سيتم استخراج ملف `app-debug.apk`.
6. **تثبيت التطبيق وإعطاء الصلاحيات آلياً باستخدام السكربت:**
   بدلاً من التثبيت اليدوي ومنح الصلاحيات المعقدة خطوة بخطوة، يُوفر المشروع سكربت أتمتة `provision_device.ps1` (بواسطة PowerShell) ليقوم بكل شيء دفعة واحدة على الأجهزة المتصلة عبر منافذ الـ USB (يجب تفعيل وضع تصحيح أخطاء USB أو ADB).
   
   - قم بتوصيل هاتف الأندرويد بالكمبيوتر عبر كابل USB ووافق على رسالة (USB Debugging).
   - تأكد من تواجد الـ APK في مساره الطبيعي `.\mdm-agent\app\build\outputs\apk\debug\app-debug.apk`.
   - افتح نافذة أوامر **PowerShell** في المجلد الرئيسي للمشروع (`Mobile_Device_Management_workspace`).
   - قم بتشغيل السكربت:
     ```powershell
     .\provision_device.ps1
     ```
   - سيقوم السكربت آلياً بالآتي:
     1. اكتشاف الأجهزة المتصلة وعرض قائمة لاختيار الهاتف المستهدف (أو التثبيت لجميع الأجهزة معاً).
     2. تثبيت ملف التطبيق `app-debug.apk`.
     3. منح جميع صلاحيات إدارة رسائل الـ SMS.
     4. منح صلاحيات الشبكات وحالة الاتصال.
     5. منح صلاحيات الوصول للملفات وقراءة الذاكرة الخارجية.
     6. إدراج التطبيق ضمن القائمة البيضاء لاستهلاك البطارية (Doze Whitelist) لضمان عمله في الخلفية باستمرار.
     7. محاولة تفعيل خدمة قراءة الإشعارات للتطبيقات الأخرى (NotificationListenerService).
     8. تشغيل تطبيق العميل (Agent Service) فوراً.
   
   > 💡 **ملاحظة:** بمجرد انتهاء السكربت وظهور رسالة النجاح، سيظهر الهاتف المتصل وتفاصيله فوراً في لوحة تحكم المشرف (Dashboard) في غضون ثوانٍ قليلة.

## 🔗 آلية الربط والتواصل في النظام (كيف تعمل كل القطع معًا؟)

نظام العمل مصمم للتشغيل المباشر والسريع عبر الخطوات المركزية المتصلة ببعضها:

1. **التعارُف الأولي للتطبيق:** بمجرد تثبيت `app-debug.apk` على الهاتف، يرسل طلب HTTP تسجيل أولي إلى `/api/v1/devices/register`.
2. **تأسيس خط الاتصال الدائم الشفاف (WebSockets):** يفتح الـ Agent الهاتف اتصالًا مستمرًا مع `/ws/agent`.
3. **وحدة التحكم (الداشبورد):** يقوم المشرف بتسجيل الدخول، ويتم إنشاء استماع WebSocket آخر على `/ws/dashboard` لانتظار التحديثات من الهواتف فورياً.
4. **توجيه وتنفيذ الأوامر (العمل الفعلي):**
   - يختار المشرف الهاتف الهدف ويضغط على زر لطلب (مثل: "قراءة معلومات الهاتف" أو "نسخ الرسائل").
   - يرسل الداشبورد الأمر للـ Backend عبر `POST`.
   - يتلقاه الـ Backend ويقوم بتمريره فوراً عبر مسار الـ WebSocket الخاص بالـ Android Agent.
   - يتلقى معالج الأوامر `CommandDispatcher.kt` بالهاتف الأمر الموجه إليه، ويقوم بتنفيذه في الخلفية.
   - يقوم الهاتف بإعادة إرسال النتائج بشكل عكسي للسيرفر، والتي بدورها تبرز للمشرف مباشرةً في شاشة الـ Dashboard.
